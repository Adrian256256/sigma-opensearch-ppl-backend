# ECS Field Mapping for Sigma Rules

## Overview

This directory contains tools and data for mapping Sigma rule fields to the Elastic Common Schema (ECS) standard. The goal is to ensure compatibility between Sigma rules and ECS-compliant SIEM platforms like OpenSearch.

---

## Files

### `checker.py`

A Python script that extracts all field names from Sigma rules in the official [SigmaHQ repository](https://github.com/SigmaHQ/sigma).

#### What it does:
1. **Recursively scans** all Sigma rule files (`.yml` and `.yaml`) in the `sigma-master/rules/` directory
2. **Parses** the `detection` section of each rule
3. **Extracts** field names, including:
   - Plain field names (e.g., `CommandLine`, `Image`)
   - Fields with modifiers (e.g., `CommandLine|contains` → extracts `CommandLine`)
   - Nested fields from complex detection logic
4. **Tracks** which rule each field was extracted from
5. **Outputs** results to two CSV files:
   - `sigma_fields.csv` - unique fields only (deduplicated and sorted)
   - `sigma_fields_with_paths.csv` - all field-rule pairs with source paths

#### Output:
- **`sigma_fields.csv`**: Contains all unique field names found across all Sigma rules (one field per line)
- **`sigma_fields_with_paths.csv`**: Contains field names with their source rule paths (multiple entries per field showing all rules using that field)

### `sigma_fields.csv`

A CSV file containing all unique Sigma field names extracted from the official Sigma rule repository.

#### Structure:
```csv
field
CommandLine
Image
ParentImage
...
```

#### Source:
- Generated by `checker.py` from the [SigmaHQ/sigma](https://github.com/SigmaHQ/sigma) repository
- Contains fields from Windows Event Logs, Sysmon, Linux audit logs, cloud platforms (AWS, Azure, GCP), network logs, and more
- One field per line, alphabetically sorted

---

### `sigma_fields_with_paths.csv`

A CSV file containing all Sigma field names along with the path to the rule file where each field was found.

#### Structure:
```csv
field,rule_path
CommandLine,windows/process_creation/proc_creation_win_susp_cmd.yml
CommandLine,windows/process_creation/proc_creation_win_powershell.yml
Image,windows/process_creation/proc_creation_win_susp_binary.yml
...
```

#### Columns:
- **`field`**: The Sigma field name
- **`rule_path`**: Relative path to the rule file (from `sigma-master/rules/`)

#### Source:
- Generated by `checker.py` from the [SigmaHQ/sigma](https://github.com/SigmaHQ/sigma) repository
- Contains multiple entries per field (one for each rule that uses it)
- Sorted alphabetically by field name, then by rule path

### `ecs_verification_results.csv`

A comprehensive mapping of Sigma fields to their ECS equivalents with confidence levels.

#### Structure:
```csv
sigma_field,ecs_field,notes
Message,message,High confidence - Official ECS core field
CommandLine,process.command_line,High confidence - Full command line
AccessMask,winlog.event_data.AccessMask,High confidence - Windows access mask
...
```

#### Columns:
- **`sigma_field`**: The original Sigma field name
- **`ecs_field`**: The recommended ECS field(s) to use
  - Single field for direct mappings (e.g., `user.name`)
  - Multiple fields separated by `|` for context-dependent mappings (e.g., `source.address | destination.address`)
- **`notes`**: Confidence level and explanation
  - **High confidence**: Direct match with official ECS fields
  - **Medium confidence**: Logical mapping but may require context or validation
  - **Low confidence**: Generic field that needs contextual interpretation

#### Important Notice:

**This file was generated with AI assistance and requires further investigation and validation.**

While the mappings are based on the official [ECS field reference documentation](https://www.elastic.co/docs/reference/ecs/ecs-field-reference), they will be reviewed and validated.
